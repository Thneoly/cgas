# 技术架构摘要（Technical Architecture Summary）

## 1. 摘要目标

本文面向工程实现，基于整理后的对话内容，抽象“文明级 Agent 生态”的技术架构骨架。重点回答：

1. 系统由哪些核心模块组成。
2. 执行如何保持确定性与可验证性。
3. 状态如何更新与提交。
4. 安全边界如何设计。
5. 如何从 MVP 逐步演化到可扩张网络。

---

## 2. 架构总览

从讨论中可以收敛为一个五层架构：

1. 接入层（Channels / Gateway）
2. 调度层（Agent Loop / Planner / Policy）
3. 执行层（AVM-like Deterministic Executor）
4. 验证层（Independent Verifier）
5. 治理与经济层（权限、激励、惩罚、参数治理）

核心原则：

- 业务灵活性在上层。
- 信任与安全约束在下层。
- 下层稳定，允许上层快速迭代。

---

## 3. 关键设计原则

### 3.1 确定性优先

执行必须满足：

- 同样输入（program + input + state_root）得到同样输出。
- 禁止隐式外部依赖（时间、随机、未声明 IO）。
- 状态变更以 diff 表示，不在执行中直接持久化。

### 3.2 验证优先

执行结果不以“节点声明”为准，而以“可重算一致”为准。

### 3.3 最小可信内核

执行内核应尽可能小（便于审计与形式化验证），复杂能力通过外围技能系统扩展。

### 3.4 分层隔离

高风险能力（文件系统、外部脚本、网络写操作）与核心执行隔离运行。

---

## 4. 核心组件说明

### 4.1 Gateway（接入网关）

职责：

- 统一接入多渠道输入（聊天平台、Webhook、API）。
- 进行会话路由、身份识别、请求标准化。
- 为后续执行生成标准请求对象。

约束：

- Gateway 不应承载高可信决策逻辑。
- 仅负责“收敛输入”，不负责“定义正确性”。

### 4.2 Agent Loop（调度与规划）

职责：

- 根据任务上下文做规划（Plan）。
- 选择工具/技能（Act）。
- 消化反馈并迭代（Observe/Reflect）。

约束：

- 可用启发式/模型驱动，但最终执行必须落入可验证执行面。
- 调度层可非确定；执行层必须确定。

### 4.3 执行内核（AVM-like Executor）

职责：

- 接收结构化 program。
- 在隔离上下文中逐条执行指令。
- 输出 state_diff、trace_hash、result_hash、gas_used。

讨论中出现的最小指令族包括：

- 状态读写：LOAD / STORE
- 算术：ADD / SUB / MUL / DIV
- 哈希：HASH
- 比较与控制流：CMP / JZ / JNZ
- 退出语义：RETURN / REVERT

设计意图：

- 指令语义简单明确。
- 控制流有限可控。
- 易于重放与审计。

### 4.4 状态层（State Store）

建议采用“执行期只读 + 输出 diff + 外部提交”的模式：

- 执行器读取旧状态。
- 产生状态差分（diff）。
- 不直接改写主状态。
- 由提交器在验证通过后 apply。

优势：

- 降低执行期破坏风险。
- 支持并行验证与回滚策略。

### 4.5 验证节点（Verifier）

职责：

- 用同样输入重跑执行。
- 比较 recomputed_result_hash 与提交结果。
- 一致则通过，不一致则拒绝。

这提供了：

- 抗伪造执行结果。
- 抗“黑箱节点”声明。
- 可争议仲裁基础。

---

## 5. 执行与验证流程（参考）

1. 请求进入 Gateway，形成标准任务。
2. Agent Loop 规划任务并生成 program。
3. 执行器在隔离上下文中运行 program。
4. 得到 state_diff + trace_hash + result_hash。
5. 验证节点独立重算。
6. 验证通过后提交状态；失败则 REVERT/拒绝。
7. 全流程记录审计轨迹。

该流程将“业务成功”与“系统可信”解耦：

- 业务可迭代。
- 可信机制保持稳定。

## 6. 数据模型建议

### 6.1 ExecutionContext

关键字段：

- pc（程序计数器）
- registers（寄存器）
- gas_used
- state_diff
- trace_hash
- status（running/success/revert）

### 6.2 StateChange

建议字段：

- key
- old_hash
- new_hash
- operation_type

### 6.3 ExecutionResult

建议字段：

- state_diff
- trace_hash
- result_hash
- gas_used
- exit_code

## 7. 安全架构摘要

### 7.1 主要攻击面

- 恶意技能（供应链风险）
- 提示注入导致越权动作
- 执行环境逃逸
- 状态伪造或回放污染
- 权限边界配置错误

### 7.2 防护策略

1. 最小权限：默认拒绝，高危显式授权。
2. 沙箱隔离：技能运行与核心执行隔离。
3. 指令白名单：核心执行仅允许受控指令。
4. 结果验证：无验证不提交。
5. 全链路审计：请求、计划、执行、提交全记录。
6. 可回滚机制：异常提交可撤销。

### 7.3 安全红线

- 禁止未验证的状态写入。
- 禁止高危技能绕过权限系统。
- 禁止在核心执行面引入不可控非确定性。

---

## 8. 可扩张性设计

### 8.1 算力分层

可按讨论思路分为：

1. Core Compute：执行/验证核心逻辑（高可信、低变更）。
2. Skill Compute：技能执行（可扩展、受隔离）。
3. AI Inference Compute：模型推理（可替换、弹性扩容）。

收益：

- 成本可控。
- 故障域分离。
- 安全与性能目标不冲突。

### 8.2 技能生态标准化

建议定义：

- Skill Manifest（能力、输入、输出、权限声明）
- 参数 schema
- 执行限制（超时、资源配额、网络策略）
- 版本与签名

---

## 9. 观测与运维

### 9.1 必要指标

- 执行成功率 / REVERT 率
- 验证一致率
- 平均 gas_used / 任务
- 高危权限调用频次
- 技能失败分布与超时分布

### 9.2 审计维度

- 谁触发了什么任务
- 计划与执行是否一致
- 哪个技能读写了哪些资源
- 状态提交是否经验证

---

## 10. 工程落地路线（建议）

### 阶段 1：最小可验证执行器

- 固化最小指令集
- 实现 deterministic executor
- 输出 trace_hash / result_hash
- 实现 verifier 重放流程

### 阶段 2：安全能力补齐

- 权限系统（RBAC/ABAC）
- 技能沙箱
- 审计日志标准
- 异常回滚

### 阶段 3：生态扩展

- Skill SDK 与发布规范
- 多节点验证
- 提案与参数治理流程

---

## 11. 非目标与边界

为了避免范围失控，建议明确非目标：

- 不在早期追求“全功能自治 Agent”。
- 不在核心执行层引入复杂业务逻辑。
- 不在验证机制未稳定前开放高危生态能力。

---

## 12. 总结

该技术架构的本质是：

**把“智能决策”与“可信执行”分层，把“灵活扩展”与“安全约束”解耦。**

只要核心执行与验证层稳定，系统就能在技能生态和业务层快速进化；反之，即使上层功能丰富，也会因不可验证和不可治理而失去长期可持续性。
