# 多角色协作运行手册（Collaboration Runbook）

## 1. 协作目标
确保多角色在 `Phase 0 ~ Phase 5` 中，以统一口径推进交付，避免“各做各的、无法汇总放行证据”。

对齐基线文档：
- `doc/review_framework_one_pager.md`
- `doc/phase_detailed_execution_blueprint.md`
- `doc/technical_architecture_implementation_plan.md`
- `doc/executive_summary.md`

## 2. 协作节奏（固定）

## 2.1 日常（Daily）
- 15 分钟站会：昨日完成、今日计划、阻塞项。
- 角色同步：PM/架构师/开发/QA/SRE/Security 至少各 1 人参与。
- 输出：当日阻塞清单与责任人。

## 2.2 周评审（Weekly）
- 评审内容：领先指标、失败分布、风险变化。
- 必备材料：测试摘要、观测摘要、部署状态、策略变更记录。
- 输出：Top 5 风险与下周优先级。

## 2.3 双周关卡（Bi-weekly Gate）
- 判定：Go / Conditional Go / No-Go。
- 依据：Entry/Exit Gate + 红线指标 + 最小样本量 + 置信约束。
- 输出：关卡结论与闭环计划。

## 2.4 月度复盘（Monthly）
- 复盘：事故归因、门禁触发次数、治理改进项关闭率。
- 输出：下月门禁规则与执行策略调整建议。

## 2.5 架构一致性检查（每周必做）
- 检查五层分层边界是否被破坏（接入/调度/执行/验证/治理）。
- 检查双平面职责是否混淆（控制平面 vs 数据平面）。
- 检查核心链路是否保持 `Executor -> Verifier -> Committer` 硬闸门。
- 检查执行阶段是否仍仅产出 `state_diff`，无越权提交。

## 3. 角色交接协议（输入/输出）

| 上游角色 | 下游角色 | 必交付输入 | 下游可开工条件 |
|---|---|---|---|
| 产品经理 | 架构师 | PRD、非目标、优先级 | 需求范围冻结、验收标准明确 |
| 架构师 | 开发工程师 | ADR、接口定义、风险取舍 | 边界清晰、失败路径明确 |
| 开发工程师 | QA | 可测试构建、测试说明、日志字段 | 关键路径可回放、指标可采集 |
| 开发工程师 | SRE | 部署说明、配置项、回滚开关 | 可灰度发布、可回滚 |
| SRE | 门禁官 | 灰度方案、回滚演练记录、稳定性评估 | 回滚可执行且发布风险可量化 |
| QA/观测工程师 | 门禁官 | TEST-MATRIX、指标证据包、报告 | 样本量达标、schema 校验通过 |
| 安全工程师 | 门禁官 | 安全评估结论、红线风险报告 | 红线风险可控或修复计划可执行 |
| 项目经理 | 门禁官 | 关卡材料包、未达标闭环计划、例外审批草案 | 资料齐全可判定 |

## 4. 标准交付物模板（角色填写）

每个角色提交结果时，统一采用以下结构：

```text
【阶段】
【角色】
【本次目标】
【输入依据】（引用文档与版本）
【产出物】
【Gate 指标映射】
【风险与缓解】
【阻塞项】
【下一步交接给】
```

提交要求（新增）：
- 必须附“指标证据四元组”：`指标值 / 时间窗口 / 样本量 / 数据来源`。
- 所有“达标”结论必须附上对应 Gate 条款编号或名称。

---

## 5. Gate 决策提示词（门禁官专用）

```text
你是发布门禁官。请基于以下材料做阶段放行审查：
1) PRD
2) ADR
3) TEST-MATRIX
4) DEPLOY-RUNBOOK
5) GATE-REPORT（json+md）

审查要求：
- 逐条核对 Entry/Exit Gate。
- 红线指标单独判定（未验证提交率、高危误放行率、未声明能力拦截率等）。
- 校验最小样本量与统计置信约束。

输出：
1) 放行结论：Go / Conditional Go / No-Go
2) 关键依据（3~5 条）
3) 未达标项（问题、责任人、修复 ETA）
4) 例外审批（审批人、范围、有效期、补偿控制）
```

---

## 6. 冲突处理机制

- 架构冲突：以 ADR 评审会决策为准，项目经理负责时限收敛。
- 指标冲突：以观测工程师统一口径为准，QA 复核。
- 安全与效率冲突：安全红线优先，允许 Conditional Go 但必须挂补偿控制。
- 进度与质量冲突：红线不妥协，范围可裁剪（降级到最小可交付）。

新增冲突优先级：
- 架构边界冲突：以技术实现方案中的层级边界为最高约束。
- 技术栈冲突：Core Runtime 优先 Rust，控制平面优先 TypeScript/Node.js。
- 性能与安全冲突：先满足安全红线，再通过分级验证与并行扩容优化性能。

---

## 7. 失败场景快速处置（门禁触发）

1. 24 小时内：冻结功能扩展，仅保留修复发布。
2. 48 小时内：完成根因归类（模型/工具/策略/基础设施）。
3. 72 小时内：提交修复方案、责任人、ETA、回归验证计划。
4. 恢复条件：最近 14 天窗口内满足门禁，并连续通过 2 次双周关卡。

---

## 8. 建议的角色编组（最小可运行）

- `Squad-A（核心执行）`：架构师 + 开发 + QA
- `Squad-B（安全治理）`：安全工程师 + 门禁官 + 项目经理
- `Squad-C（运营发布）`：SRE + 观测工程师 + PM

三组并行，项目经理负责跨组依赖与 Gate 节点合并。

---

## 9. 关卡前置检查清单（Pre-Gate Checklist）

在进入双周关卡前，项目经理必须确认：
- [ ] 交付附件齐全：PRD/ADR/TEST-MATRIX/DEPLOY-RUNBOOK/GATE-REPORT
- [ ] `gate-report.json` 已通过 schema 校验
- [ ] 红线指标单独报告且证据可追溯
- [ ] 高危相关指标样本量达标（不足仅允许修复发布）
- [ ] 安全评审已给出是否可放行结论
- [ ] 回滚路径已演练（至少预发）

---

## 10. 阶段判定口径（执行版）

- `Go`：红线与阶段 Exit Gate 全满足，无关键阻塞。
- `Conditional Go`：存在非红线未达标项，已明确责任人、ETA、补偿控制、有效期。
- `No-Go`：任一红线未满足、证据不足或关键附件缺失。

判定输出必须包含：
1) 结论
2) 依据指标（3~5）
3) 未达标项闭环计划
4) 例外审批记录（如有）
